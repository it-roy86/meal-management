## 2. 배경 및 문제점

- 어머니가 구내식당 운영 중 매일 두 회사의 식사 명단을 종이에 기록
- 기록한 종이를 사진으로 보내면 아드님이 엑셀로 정리
- 월말마다 회사별 정산 후 이메일로 청구서 발송

**현재 흐름:** 종이 기록 → 사진 전송 → 엑셀 수동 입력 → 월말 정산

**목표 흐름:** 앱 직접 입력 → 자동 정리 → 월말 자동 정산

---

## 3. 사용자 및 권한

| 역할 | 대상 | 권한 |
|------|------|------|
| ADMIN | 아드님 (개발자) | 전체 데이터 관리, 설정, 정산, 사용자 계정 관리 |
| OPERATOR | 어머니 (식당 운영자) | 일일 식사 인원 입력, 당일 기록 수정 |
| VIEWER | 경리담당자 (회사별) | 자기 회사 데이터만 읽기 전용 조회 |

> VIEWER는 자기 회사 데이터만 접근 가능 (대박유통 담당자 → 대박유통 데이터만)

## 5. 핵심 기능

### 기능 01. 일일 식사 인원 입력 ★ 핵심
- 날짜 선택 시 회사/팀별 행 자동 표시
- 중식/석식 인원 숫자 입력 → 소계 자동계산
- 모바일 친화적 UI (어머니가 스마트폰으로 입력)
- 입력/수정 방식: 같은 날짜+팀 데이터는 업데이트 방식으로 처리

### 기능 02. 현황 조회
- 날짜 범위, 회사별 필터링
- VIEWER 역할은 자기 회사 데이터만 자동 필터 적용

### 기능 03. 월별 정산 ★ 중요
- 월 + 회사 선택
- 팀별 중식/석식 인원 및 금액 자동 집계
- 총 식사인원 합계, 총 금액 합계 표시
- 엑셀 파일 다운로드 기능

### 기능 04. 설정 관리 (ADMIN 전용)
- 회사 추가/수정 (회사명, 사업자번호, 담당자 이메일)
- 팀 추가/수정 (팀명, 중식/석식 단가)
- 사용자 계정 생성 및 역할 부여
- 소프트 딜리트 적용 (삭제 대신 is_active = false 처리)

### 기능 05. 메일 발송 (추후 개발)
- 월별 정산 데이터를 회사 경리담당자에게 이메일 발송
- contact_email 컬럼 활용
- API 구조는 메일 기능 추가에 용이하게 미리 설계

## 9. 설계 주요 결정사항

- **소프트 딜리트 적용**: 팀/회사 삭제 시 DELETE 대신 `is_active = false` 처리 → 과거 정산 데이터 보호
- **단가 스냅샷 저장**: `meal_record.total_amount`에 입력 시점 단가 기반 금액 저장 → 나중에 단가 변경되어도 과거 정산 영향 없음
- **입력자 추적**: `created_by`로 누가 입력했는지 추적 가능
- **VIEWER 데이터 격리**: `user.company_id`로 경리담당자별 접근 회사 제한
- **중복 방지 전략**: UNIQUE 제약 대신 입력 화면에서 날짜+팀 기준으로 기존 데이터 불러와 수정하는 방식 채택
